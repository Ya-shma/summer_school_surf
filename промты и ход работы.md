# Скалодром "Вертикаль"

Заказчик: Оля, владелица скалодрома «Вертикаль» (бизнес работает третий год, площадка расположена в бывшем складском ангаре).  
Ключевая проблема (боль): Оля полностью «зашилась» с ручным учётом записей через Telegram и бумажную тетрадь. В часы пик это превращается в ад: возникают двойные брони, путаница с инструкторами и ситуация, когда клиент приходит, а группа уже занимается.  
Главная цель: Запустить мобильное приложение, чтобы клиенты записывались на тренировки самостоятельно, а она могла просто контролировать процесс и смотреть расписание.  
Ключевые ограничения и условия:

* Жёсткие сроки: Дедлайн — начало нового сезона (через \~2 месяца).  
* Ограниченный бюджет: Требуется строгий MVP-подход. Нужно сделать только самое необходимое для снятия боли, а все «навороты» отложить на потом.  
* Финансы: На старте оплата принимается только офлайн (наличные или перевод на карту). В онлайн-эквайринг на первом этапе вкладываться не готова.  
* Коммуникация: Для общения с клиентами и сбора обратной связи использует Telegram (встроенные чаты в приложении не нужны).

Контекст её бизнеса (клиенты и процессы):

* Аудитория: Новички (болдеринг с инструктажем) и опытные (трассы с верёвкой). Также ходят дети с 6 лет, но строго в сопровождении родителей.  
* Команда: 5 основных инструкторов \+ подработчики в сезон.  
* Инвентарь: 20 комплектов снаряжения для проката.  
* Специфика: Бывает профилактика (перекрутка трасс), из\-за которой приходится экстренно отменять тренировки и предупреждать людей. Также есть проблема неявок и поздних отмен, из\-за которых места простаивают.

# Тестирование

# Баг 1: Клиенты с принятыми правилами ТБ не могут записаться

## Симптом  
При попытке записаться на тренировку возвращается ошибка `400 Bad Request` с кодом `safety_rules_not_accepted`, даже если клиент уже принял правила техники безопасности через модалку `SafetyRulesModal`. В логах бэкенда видно, что `POST /api/auth/profile` возвращает `200 OK`, но при следующем запросе `GET /api/auth/me` поле `safety_rules_accepted` остаётся `false`.

## Требование  
- **FR-51** — при первой записи клиент должен принять правила ТБ, флаг фиксируется в профиле  
- **FR-10** — запись на выбранный слот  
- **US-9** — согласие с правилами и ТБ при первой записи  
- **UC-3** — Запись на тренировку (шаг 4)  
- **BS-002** — Согласие с правилами и ТБ

## Промпт ИИ  
- При записи на тренировку возвращается ошибка 400 safety_rules_not_accepted, даже если клиент уже принял правила. В логах видно, что POST /api/auth/profile возвращает 200, но флаг safety_rules_accepted остаётся False. Исправь.

**Решение:** В схеме `ProfileUpdate` отсутствовало поле `safety_rules_accepted`. Pydantic по умолчанию игнорирует неизвестные поля, поэтому при отправке `{safety_rules_accepted: true}` флаг не сохранялся в БД. Нужно добавить поле в схему и обработку в роуте `/auth/profile`.

# Баг 3: Запись не работает

## Симптом  
При попытке записаться на тренировку возвращается ошибка `400 Bad Request` с сообщением «Idempotency-Key обязателен». В DevTools Network видно, что запрос `POST /api/bookings/` уходит без заголовка `Idempotency-Key`, хотя в коде `frontend/src/api.js` есть функция `generateIdempotencyKey()`.

## Требование  
- **NFR-27** — API мутирующих операций должен поддерживать идемпотентность через заголовок `Idempotency-Key`  
- **BR-1** — устранение двойных броней  
- **US-24** — защита от двойной брони при сбое сети  
- **UC-3** — Запись на тренировку (шаг 7)  
- **LOGIC-008** — Идемпотентность мутаций

## Промпт ИИ  
- При записи на тренировку возвращается ошибка 400 "Idempotency-Key обязателен".  
Исправь.

**Решение:** Функция `crypto.randomUUID()` не работает в не-secure контексте (HTTP вместо HTTPS). Когда фронтенд открывается по IP, `crypto.randomUUID()` возвращает `undefined`, и заголовок не добавляется. Нужно добавить fallback-генерацию UUID v4 для HTTP-контекста.
