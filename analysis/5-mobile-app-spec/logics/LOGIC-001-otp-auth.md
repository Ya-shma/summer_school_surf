# LOGIC-001 — OTP-авторизация

## Цель
Обеспечить безопасную регистрацию и вход по SMS OTP.

## Связь с требованиями
- **FR:** FR-1, FR-2, FR-43, FR-50
- **NFR:** NFR-18 (сессия JWT), NFR-19 (антифрод OTP)
- **US:** US-1
- **UC:** UC-1

---

## Входные данные
- Номер телефона.
- Согласие на ПДн (FR-50).
- OTP-код.

## Выходные данные
- `access_token`, `refresh_token`.
- Профиль (или `null` для нового клиента).

---

## Алгоритм
1. Клиент вводит телефон, ставит галочку ПДн.
2. Клиент тапает «Получить код» → `POST /auth/otp/request {phone, consent_pd=true}`.
3. Сервер отправляет SMS с 4–6-значным кодом (TTL 5 мин, NFR-19).
4. Клиент вводит код → `POST /auth/otp/verify {phone, code}`.
5. При успехе — возвращает пару токенов (access + refresh).
6. Если у клиента нет профиля (новый номер) — запрашивает имя (обяз.), возраст и ДР (опц.).
7. Токены сохраняются в Keychain/Keystore (NFR-18).
8. Push-токен регистрируется на бэкенде (`POST /push/token`).

## Rate-limit (NFR-19)
- ~1 запрос кода в 30–60 с на номер.
- ~5 запросов в час.
- После 3 неверных попыток ввода кода — блокировка на 15 минут (с таймером).
- При повторной отправке старый код аннулируется, новый имеет свой TTL 5 минут.

## Refresh-flow (NFR-18)
- При `401 Unauthorized` от API → `POST /auth/refresh {refresh_token}`.
- При успехе — новая пара токенов, повтор исходного запроса.
- При неуспехе — очистка Keychain/Keystore → SCR-001.

## Обработка ошибок
| Код | HTTP | UX-реакция |
|---|---|---|
| Недоставка SMS | — | «Не удалось отправить код, попробуйте позже» |
| `invalid_code` | 400/401 | «Неверный код, попробуйте ещё раз» |
| `429` + `retry_after` | 429 | «Слишком много попыток, попробуйте через N минут» |
| Нет сети | — | Баннер «Нет соединения» + Retry |
