Экран: Вход / регистрация
Связанный файл: `design-brief.md`
UC: UC-1 (Регистрация и вход по SMS OTP)
Порядковый номер: 1

Цель
Максимально быстро ввести клиента в сессию.

Связь с требованиями
FR: FR-1, FR-2, FR-43, FR-50
NFR: NFR-3 (лёгкая регистрация), NFR-19 (антифрод OTP)
US: US-1
UC: UC-1

Структура экрана
1. Ввод телефона: поле с маской `+7 (___) ___-__-__`, кнопка «Получить код» (изначально disabled).
2. Чекбокс ПДн: «Согласен на обработку персональных данных» (обязательный для активации кнопки).
3. Экран OTP: поле ввода 4–6 цифр, таймер повторной отправки (30–60 с), ссылка «Не пришёл код?». Поддержка автозаполнения кода через WebOTP API (при поддержке браузером) или ручной ввод.
4. Для новых пользователей: краткая форма профиля (имя обязательно, возраст и день рождения опционально).

Состояния
| Состояние | Поведение |
|---|---|
| Default | Поле телефона активно, чекбокс ПДн не отмечен, кнопка disabled |
| Загрузка (SMS) | Спиннер на кнопке, поле телефона disabled |
| Ввод кода | Фокус на первом поле, клавиатура numeric, автозаполнение (WebOTP) или ручной ввод |
| Таймер повторной отправки | Кнопка/ссылка disabled, обратный отсчёт |
| Блокировка (после 3 ошибок) | Сообщение «Попробуйте через 15 мин», таймер, поля disabled |
| Offline | Баннер «Нет соединения», действия заблокированы |

Заметки для дизайнера
- Чекбокс ПДн должен быть явно связан с политикой конфиденциальности (ссылка).
- При ошибке `invalid_code` подсвечивать поле кода, не сбрасывать фокус, показывать текст «Неверный код, попробуйте ещё раз».
- При `429` показывать таймер до повторной попытки, блокировать ввод.
- Максимум 2 шага для существующего клиента, 3 шага для нового (DG-1, DG-3).
- Не блокировать экран при недоставке SMS — давать явный путь повторной отправки или обращения в поддержку.
