Скоуп — клиентское веб-приложение и API для него. Бизнес-цели, обслуживаемые существующей инфраструктурой (картина для владельца, интерфейс инструктора, админка, CRUD расписания), в этот список не входят.

**Роли в текущей поставке.** В текущую поставку входит только роль Клиент (самостоятельная запись на тренировки через веб-приложение). Интерфейсы инструктора и администратора/владельца — вне скоупа текущей поставки: они обеспечиваются существующей инфраструктурой. Это осознанное решение в рамках MVP-подхода и жёсткого срока: продукт фокусируется на клиентском сценарии, который снимает основную боль (хаос записи и двойные брони), а серверные/ролевые функции остаются на стороне существующего бэкенда.

**Статусы бронирования (единая терминология для BR).** Бэкенд ведёт бронь в одном из статусов; приложение отображает и обрабатывает их согласно контракту API:

- `booked` — активная бронь.
- `cancelled_by_client` — отменена клиентом заблаговременно (≥ 6 ч до старта).
- `cancelled_late` — поздняя отмена клиентом (< 6 ч до старта), квалифицируется как нарушение.
- `no_show` — неявка (бронь не отменена, но клиент не пришёл), квалифицируется как нарушение.
- `cancelled_by_gym` — отменена скалодромом (профилактика, проверка снаряжения).
- `attended` — факт посещения подтверждён (бронь была активна в момент начала тренировки и не перешла в `cancelled_late` / `no_show`).

## Ограничения верхнего уровня

- **Срок и бюджет.** Жёсткая цель — начало следующего сезона (~2 месяца); бюджет ограничен → строго MVP-подход, «навороты» откладываются. (Бриф; Ответы заказчика, Блок 9, № 3)
- **Оплата офлайн на старте.** Онлайн-оплата вне границ скоупа, цена фиксированная (1500 ₽). (Ответы заказчика, Блок 4, № 6; Блок 3, № 4)
- **Responsive / mobile-first.** Клиентское веб-приложение рассчитано на использование с телефона через браузер, но корректно работает и на десктопе; данные получает из существующего бэкенда через API. (Ответы заказчика, Блок 8, № 1)
- **Вне скоупа.** Функции инструктора/владельца, расписание-CRUD и админка обеспечиваются существующей инфраструктурой. Онлайн-оплата, расширенные механики лояльности (скидки, бесплатные тренировки), публичный рейтинг инструкторов, текстовые комментарии к оценкам, SMS/WhatsApp-каналы, встроенный чат поддержки — направления развития (V2+). (Описание домена → «Границы скоупа»)
- **Оценки инструкторов — приватные.** Оценки видны только владельцу в существующей админке; публичный рейтинг клиентам не показывается. Это осознанное решение, согласованное с заказчиком: чтобы избежать «охоты на звёзд» и обид на новеньких. (Ответы заказчика, Блок 7, № 1; Описание домена → «Оценки инструктора»)
- **Лимиты по формату — жёсткие.** 8 для новичков, 16 для опытных. Инструктор менять лимит не может. (Ответы заказчика, Блок 4, № 1)
- **Одна бронь = один клиент.** Записать нескольких человек в одной брони нельзя. Исключение — случай «родитель + ребёнок», который оформляется двумя отдельными бронями на один и тот же слот (см. BR-13). (Ответы заказчика, Блок 4, № 3; Блок 1, № 3)
- **Push — единственный канал уведомлений.** Используются браузерные push через Web Push API. SMS не используются даже для критичных событий (отмена скалодромом, список ожидания). (Ответы заказчика, Блок 8, № 1–3)
- **Список ожидания — Alert List, не автобронь.** Уведомление не резервирует место за клиентом автоматически; клиент сам записывается после получения push. (Ответы заказчика, Блок 6, № 3)
- **Факт посещения определяется бэкендом.** Статус `attended` выставляется автоматически: если на момент начала тренировки бронь активна (не `cancelled_*`, не `no_show`), клиент считается посетившим. Ручная отметка инструктора в клиентском веб-приложении не требуется — отметка о приходе ведётся в существующей инфраструктуре (вне скоупа). (Ответы заказчика, Блок 7, № 2)
- **Счётчик нарушений сбрасывается после посещения.** 3 нарушения считаются «подряд» — без промежуточных успешных посещений (статус `attended`). После посещения счётчик обнуляется. (Ответы заказчика, Блок 6, № 2)
