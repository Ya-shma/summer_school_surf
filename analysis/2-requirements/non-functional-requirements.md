**Целевая платформа (R-002).** Продукт — нативное/гибридное мобильное приложение для iOS и Android, распространяемое через App Store и Google Play. Требования к push, хранению токена, сессии и a11y сформулированы исходя из этого решения.

## Надёжность и целостность данных

| ID | Категория | Требование | Приоритет | Источник |
|---|---|---|---|---|
| NFR-8 | Надёжность / Целостность данных | При одновременной записи нескольких клиентов на один слот исключаются двойные брони и овербукинг: количество занятых мест и выданных прокатных комплектов снаряжения никогда не превышает лимитов слота (8/16 мест, 20 комплектов проката). Это критическое требование (главная боль заказчика, BR-1). | Высокий (Must) | бриф; описание домена; BR-1; BR-3 |
| NFR-9 | Надёжность / Целостность данных | Состояние записей и свободных мест/проката остаётся согласованным при сбоях и параллельных операциях (отмена освобождает место и прокатный комплект корректно, без потери или задвоения данных). | Высокий | бриф; описание домена |
| NFR-24 | Надёжность / Offline | Просмотр уже загруженных данных (списка слотов, своих броней) доступен офлайн из кэша с явной пометкой «данные могли устареть». Мутации (создание/отмена брони, оценка, добавление в Alert List) офлайн запрещены — показывается понятное сообщение о необходимости подключения. Единый паттерн Error/Retry для всех сетевых ошибок. Таймаут сетевого запроса — ~10 с. При неопределённом результате мутирующей операции (таймаут/обрыв) клиент повторяет запрос с тем же Idempotency-Key (см. NFR-27), что исключает дубли. | Средний | R-020 |
| NFR-27 | Надёжность / Идемпотентность | API мутирующих операций (`createBooking`, `cancelBooking`, `joinWaitlist`, `leaveWaitlist`, `rateInstructor`) должен поддерживать идемпотентность через заголовок `Idempotency-Key`. Клиент генерирует уникальный ключ (UUID v4) для каждой операции на стороне приложения и повторяет запрос с тем же ключом при неопределённом результате (таймаут, `5xx`, обрыв соединения). Сервер гарантирует, что повторный запрос с тем же ключом в окне идемпотентности (не менее 24 ч) не создаёт дублирующую сущность и возвращает тот же ответ, что и первый успешный вызов. Ключ передаётся в заголовке `Idempotency-Key`; при его отсутствии сервер возвращает `400 Bad Request` для мутирующих эндпоинтов. Это требование напрямую защищает BR-1 (устранение двойных броней) в условиях нестабильной мобильной сети. | Высокий (Must) | BR-1; NFR-8; NFR-24 |
| NFR-31 | Надёжность / Актуальность данных | При открытии карточки слота приложение запрашивает актуальные данные с бэкенда (не полагается на кэш) — это критично для корректного отображения числа свободных мест и прокатного фонда перед записью. В списке слотов допустимо использовать кэш с пометкой «данные могли устареть» и кнопкой «Обновить». При получении `409 Conflict` / `slot_full` на `createBooking` клиент показывает понятное сообщение «места закончились» и обновляет карточку слота. Это MVP-компромисс между реалтаймом и простотой реализации. | Средний | BR-1; NFR-8 |
