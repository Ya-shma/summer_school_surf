Скоуп: клиентское веб-приложение и API для него. Слоты, форматы и инструкторы поступают из существующего бэкенда через API (read-only). Флаги допуска, статусы лояльности, счётчики нарушений и блокировки ведутся в существующей инфраструктуре; веб-приложение их потребляет.

## UC-1: Регистрация и вход по SMS OTP

**Актёры:** Клиент
**Предусловия:** У клиента есть действующий номер телефона; SMS-провайдер доступен.
**Триггер:** Клиент впервые открывает веб-приложение или хочет войти в существующий аккаунт.

**Бизнес-решения:**
- Регистрация только по телефону (BR-15, FR-1, FR-2, FR-43). Email и соцсети не поддерживаются.
- Согласие на обработку ПДн (FR-50, NFR-20). Обязательно при регистрации.
- Антифрод OTP (NFR-19). Rate-limit на отправку кода, TTL 5 минут, блокировка после 3 неверных попыток на 15 минут.
- Сессия JWT (NFR-18). Access + refresh токены хранятся в HttpOnly cookies (или secure localStorage с учётом XSS-защиты).

**Основной поток:**
1. Клиент открывает веб-приложение → попадает на экран входа.
2. Вводит номер телефона (с маской `+7`).
3. Ставит галочку «Согласен на обработку персональных данных» (FR-50). Без галочки кнопка «Получить код» неактивна.
4. Нажимает «Получить код».
5. Система отправляет запрос на SMS OTP. Клиент получает SMS с 4–6-значным кодом (TTL 5 минут, NFR-19).
6. Клиент вводит код.
7. Система отправляет код на верификацию. При успехе — возвращает пару токенов (access + refresh).
8. Если у клиента нет профиля (новый номер) — система запрашивает имя (обязательно), возраст и день рождения (опционально, FR-47).
9. Клиент вводит данные → профиль создан, сессия активна.
10. Веб-приложение переходит на главный экран — список слотов (FR-9).

**Альтернативные потоки:**
- **A1. Повторная отправка кода.** Клиент не получил SMS → нажимает «Отправить код повторно» не ранее чем через 30–60 с (NFR-19). Старый код аннулируется, новый имеет свой TTL 5 минут.
- **A2. Существующий клиент.** Если номер уже зарегистрирован — после ввода OTP шаг 8 пропускается, клиент сразу попадает в список слотов.
- **A3. Восстановление доступа.** Клиент сменил номер → регистрируется заново по новому номеру. Старый аккаунт остаётся; привязку к новому номеру выполняет владелец вручную через админку.

**Исключения / ошибки:**
- **E1. Недоставка SMS.** Провайдер вернул ошибку → показывается сообщение «Не удалось отправить код, попробуйте позже» (NFR-23).
- **E2. Неверный код.** Сервер вернул `invalid_code` (400/401) → поле OTP остаётся активным, показывается «Неверный код, попробуйте ещё раз». После 3 неверных попыток — блокировка на 15 минут с таймером (NFR-19).
- **E3. Rate-limit исчерпан.** Сервер вернул `429` с `retry_after` → показывается сообщение «Слишком много попыток, попробуйте через N минут».
- **E4. Нет сети.** Мутация входа офлайн запрещена → показывается понятное сообщение о необходимости подключения.

**Постусловия:**
- При успехе: сессия активна, push-подписка зарегистрирована на бэкенде, клиент на главном экране.
- При ошибке: сессия не создана, клиент остаётся на экране входа.

## UC-2: Просмотр и фильтрация списка слотов

**Актёры:** Клиент
**Предусловия:** Клиент авторизован.
**Триггер:** Клиент хочет найти подходящую тренировку.

**Бизнес-решения:**
- Дефолтный период 7 дней (R-027, FR-9).
- Приоритетная запись постоянных (BR-10, FR-52). Постоянные видят слот за 2 дня, остальные — за 1 день.
- Семантика фильтров (FR-38). Внутри группы — ИЛИ, между группами — И.

**Основной поток:**
1. Клиент открывает список слотов. По умолчанию показываются слоты на ближайшие 7 дней (`date_from = now`, `date_to = now + 7 дней`), отсортированные по времени старта (по возрастанию).
2. Заполненные (`free_seats = 0`) и отменённые скалодромом слоты отображаются с пометкой «Мест нет» / «Отменена скалодромом» и неактивной CTA записи.
3. Для слотов, которые ещё не открыты для категории клиента (FR-52), показывается пояснение «Откроется для записи через N дней» и неактивная кнопка записи.
4. (Опц.) Клиент применяет фильтры: дата/период, формат (новичковый/опытный), инструктор, опция «только свободные».
5. Система показывает отфильтрованный список с ключевыми параметрами каждого слота: время (крупно), формат, свободно мест, цена.
6. Клиент нажимает на слот → открывается карточка (FR-9a) с деталями: дата/время, формат, описание, инструктор, цена (1500 ₽), всего/свободно мест, свободный прокатный фонд, адрес, чек-лист «что взять», ссылка на ТБ.

**Альтернативные потоки:**
- **A1. Сброс фильтров.** Клиент очищает фильтры → возвращается дефолтная выдача.
- **A2. Только свободные.** Клиент включает фильтр `only_available=true` → заполненные/отменённые слоты скрываются.
- **A3. Семантика фильтров (FR-38).** Внутри одной группы значения комбинируются по ИЛИ (формат, инструктор — массивы), между группами — по И. Границы периода дат включительные.
- **A4. Пустой справочник инструкторов.** Если список инструкторов пуст, фильтр по инструктору скрывается.
- **A5. Постоянный клиент.** Если у клиента флаг `is_permanent=true` (FR-59), слоты открываются для записи за 2 дня до старта (BR-10).
- **A6. Актуализация карточки (NFR-31).** При открытии карточки слота веб-приложение запрашивает актуальные данные с бэкенда (не из кэша).

**Исключения / ошибки:**
- **E1. Нет слотов под условия.** Показывается empty state «Пока нет доступных тренировок» с подсказкой изменить/сбросить фильтры.
- **E2. Нет расписания вовсе.** В горизонте выдачи нет ни одного предстоящего слота → empty state.
- **E3. Просмотр офлайн (NFR-24).** При отсутствии сети показывается ранее загруженный кэш списка (через Service Worker / Cache API) с пометкой «данные могли устареть» и кнопкой «Обновить».

**Постусловия:** Клиент видит актуальный (или помеченный как устаревший) список предстоящих слотов; данные расписания не изменяются.

## UC-3: Запись на тренировку

**Актёры:** Клиент
**Предусловия:** Клиент авторизован; в расписании есть слот со свободными местами; клиент не заблокирован из-за нарушений.
**Триггер:** Клиент хочет забронировать место на тренировке.

**Бизнес-решения:**
- Одна бронь = один участник (BR-13, FR-54). Родитель записывает ребёнка отдельной бронью с указанием возраста.
- Гейткинг по формату (BR-4, FR-55). Без флага `rope_access` запись на опытный слот запрещена.
- Контроль проката (BR-3, FR-14, FR-15). Прокат резервируется немедленно; при нулевом фонде запись с прокатом запрещена.
- Идемпотентность (NFR-27). Клиент передаёт `Idempotency-Key` при создании брони.
- Согласие с правилами ТБ (FR-51). При первой записи — модальное окно с галочкой.

**Основной поток:**
1. Клиент открывает карточку слота (UC-2, шаг 6).
2. Система проверяет:
   - слот не отменён и не стартовал;
   - у клиента есть флаг `rope_access` (если формат «опытный»);
   - клиент не заблокирован из-за нарушений;
   - есть свободные места и (если нужен прокат) свободные комплекты.
3. Клиент нажимает «Записаться».
4. Если это первая запись в жизни клиента — показывается модальное окно «Ознакомлен с правилами и техникой безопасности» с галочкой (FR-51). Без согласия кнопка подтверждения неактивна. Флаг фиксируется в профиле.
5. Клиент выбирает вариант снаряжения: «свои скальники» или «нужно прокатное снаряжение» (FR-11).
6. Если клиент записывает ребёнка (с 6 лет) — выбирает опцию «Записать ребёнка» и указывает возраст (FR-54).
7. Клиент подтверждает запись. Веб-приложение формирует и передаёт `Idempotency-Key` (UUID v4) в заголовке (NFR-27).
8. Система проверяет лимиты:
   - `free_seats ≥ 1` (лимит формата: 8 для новичков, 16 для опытных — FR-13);
   - если выбран прокат — `free_rental_boards ≥ 1` (FR-14).
9. Сервер создаёт бронь, уменьшает счётчики мест и (если применимо) прокатного фонда. Бронь получает статус `booked`.
10. Система показывает подтверждение с ценой (1500 ₽ + тариф проката, если выбран; FR-45) и добавляет запись в «Мои бронирования».

**Альтернативные потоки:**
- **A1. Свои скальники.** Прокатный фонд не затрагивается; уменьшается только число мест в группе. Клиент может записаться даже при нулевом фонде проката (BR-3).
- **A2. Запись ребёнка.** Клиент выбирает «Записать ребёнка» → указывает возраст (обязательное поле, ≥ 6 лет). Создаётся отдельная бронь на того же клиента-родителя с пометкой возраста. Родитель, желающий прийти с ребёнком, оформляет две последовательные брони на один и тот же слот (BR-13).
- **A3. Повторная запись с тем же Idempotency-Key.** При сетевом сбое клиент повторяет запрос с тем же ключом → сервер возвращает тот же ответ, дубль брони не создаётся (NFR-27).

**Исключения / ошибки:**
- **E1. Нет мест в группе.** Сервер вернул `slot_full` (409) → показывается актуальное число свободных мест, карточка обновляется.
- **E2. Нет прокатных комплектов.** Если выбран прокат, а `free_rental_boards = 0` → кнопка «Записаться» неактивна (FR-15); если клиент уже нажал — сервер вернёт ошибку, веб-приложение предлагает выбрать «свои скальники».
- **E3. Гейткинг по формату.** Клиент без флага `rope_access` пытается записаться на опытный слот → кнопка «Записаться» неактивна, текст «сначала пройдите вводную тренировку для новичков» (FR-55). Если каким-то образом запрос ушёл — сервер вернёт `rope_access_required` (403).
- **E4. Клиент заблокирован.** Сервер вернул `client_blocked` (403) с датой окончания блокировки → показывается сообщение «Вы заблокированы до N из-за нарушений. Обратитесь к владельцу».
- **E5. Двойная бронь на тот же слот.** Сервер вернул `double_booking` (409) → переход в «Мои бронирования».
- **E6. Слот отменён или стартовал.** Сервер вернул `slot_cancelled` (410) или `slot_started` (422) → запись отклоняется, CTA записи убирается.
- **E7. Сетевой сбой / неопределённый результат (NFR-24, NFR-27).** Мутация офлайн запрещена — при отсутствии сети кнопка подтверждения недоступна. При таймауте/обрыве клиент повторяет запрос с тем же `Idempotency-Key`; дубль не создаётся.
- **E8. Цена изменилась.** Веб-приложение использует `price_total` из ответа API (FR-45) — сервер источник истины.

**Матрица ошибок (дополнение к UC-1):**

| Код | HTTP | Обязательные details | UX-реакция | Retry |
|---|---|---|---|---|
| slot_full | 409 | available_seats, available_rental_boards | Показать актуальные свободные места/прокат, обновить карточку | Нет (изменить выбор) |
| double_booking | 409 | booking_id | Сообщить, что бронь уже есть; перейти в «Мои бронирования» | Нет |
| rope_access_required | 403 | slot_id, format | «Сначала пройдите вводную тренировку для новичков» | Нет |
| client_blocked | 403 | blocked_until, reason | «Вы заблокированы до N» | Нет |
| rental_unavailable | 409 | available_rental_boards | «Проката нет, выберите свои скальники» | Нет |
| slot_cancelled | 410 | slot_id, reason | Убрать CTA записи, показать причину отмены | Нет |
| slot_started | 422 | slot_id, start_at | «Тренировка началась/прошла» | Нет |
| safety_rules_not_accepted | 400 | — | Вернуть клиента к шагу согласия с ТБ | Да |
| child_age_invalid | 400 | min_age | «Возраст ребёнка должен быть ≥ 6 лет» | Да |

**Постусловия:**
- При успехе: создана бронь (статус `booked`); счётчики слота и (если применимо) прокатного фонда уменьшены; запись видна в «Мои бронирования».
- При ошибке: состояние слота не изменилось.

## UC-4: Отмена записи

**Актёры:** Клиент
**Предусловия:** У клиента есть предстоящая запись (статус `booked`); тренировка ещё не стартовала.
**Триггер:** Клиент решает отменить участие.

**Бизнес-решения:**
- Полная отмена (FR-16). Отмена возможна только целиком — освобождается место и прокатный комплект этой брони.
- Порог 6 часов (BR-5, FR-17, FR-18). Ранняя отмена (≥ 6 ч) — бесплатно; поздняя (< 6 ч) — нарушение, кнопка «Отменить» заблокирована.
- Источник истины — сервер (FR-17). Время и статус отмены определяются сервером.

**Основной поток:**
1. Клиент открывает «Мои бронирования» и выбирает предстоящую запись.
2. Система запрашивает у сервера актуальный статус брони и флаг `can_cancel` (FR-18). Сервер вычисляет `can_cancel` на основе времени старта и правила 6 часов.
3. Если `can_cancel = true` — кнопка «Отменить» активна.
4. Клиент нажимает «Отменить».
5. Сервер определяет время до старта. Ранняя отмена (≥ 6 ч до старта включительно):
   - бронь переходит в статус `cancelled_by_client`;
   - место и прокатный комплект (если был) возвращаются в слот;
   - уведомляется первый клиент из Alert List этого слота (FR-61).
6. Система показывает подтверждение отмены.

**Альтернативные потоки:**
- **A1. Поздняя отмена (< 6 ч до старта).** Сервер вернул `can_cancel = false`. Кнопка «Отменить» неактивна. Клиенту показывается пояснение политики 6 часов и кнопка «Написать Оле» (FR-16a), открывающая чат в Telegram через веб-ссылку.
- **A2. Расхождение клиентского прогноза и сервера.** Клиент показал «ещё можно отменить», а сервер вернул `can_cancel = false` → применяется и отображается серверный статус.

**Исключения / ошибки:**
- **E1. Тренировка уже стартовала/прошла.** Сервер вернул `slot_started` (422). Кнопка отмены недоступна.
- **E2. Бронь уже отменена.** Сервер вернул `already_cancelled` → показывается текущий статус, повторная отмена не выполняется.
- **E3. Отмена офлайн / неопределённый результат (NFR-24).** Мутация офлайн запрещена — кнопка недоступна. При неопределённом результате клиент повторяет запрос с тем же `Idempotency-Key`; финальный статус берётся с сервера.

**Постусловия:** Статус брони обновлён (`cancelled_by_client` / `cancelled_late`). При ранней отмене место и прокат снова доступны другим клиентам; первый в Alert List получает push.

## UC-5: Подписка на список ожидания (Alert List)

**Актёры:** Клиент
**Предусловия:** Клиент авторизован; слот заполнен (`free_seats = 0`) и не отменён.
**Триггер:** Клиент хочет узнать, если на заполненной тренировке освободится место.

**Бизнес-решения:**
- Alert List, не автобронь (BR-6, FR-53, FR-61). Уведомление не резервирует место — клиент сам записывается после push.
- Один клиент — одна подписка на слот. Повторная подписка запрещена.

**Основной поток:**
1. Клиент открывает карточку заполненного слота (UC-2).
2. Система показывает кнопку «Уведомить, если появится место» (FR-53).
3. Клиент нажимает кнопку.
4. Сервер добавляет клиента в Alert List слота.
5. Кнопка меняется на «Вы в списке ожидания» с возможностью отписаться.

**Альтернативные потоки:**
- **A1. Отписка от Alert List.** Клиент нажимает «Отписаться» → сервер удаляет клиента из очереди.
- **A2. Освобождение места.** При отмене другой брони (UC-4) сервер отправляет push первому клиенту в Alert List (FR-61). Клиент получает push и самостоятельно записывается через веб-приложение (UC-3). Место не резервируется автоматически.
- **A3.** Если первый клиент из Alert List не может записаться (например, уже имеет бронь на пересекающийся слот, заблокирован, или слот снова заполнился до того, как он успел записаться), уведомление считается доставленным, но место переходит к следующему клиенту в очереди. Повторный push для того же освободившегося места не отправляется — клиент должен следить за слотом вручную.

**Исключения / ошибки:**
- **E1. Клиент уже в списке.** Сервер вернул `waitlist_already_joined` (409) → кнопка сразу показывает «Вы в списке ожидания».
- **E2. Слот уже не заполнен.** При попытке подписаться на слот, где уже есть места → сервер вернёт ошибку, веб-приложение переключает UI на стандартную кнопку «Записаться».
- **E3. Слот отменён.** Сервер вернул `slot_cancelled` (410) → кнопка Alert List скрывается.

**Постусловия:**
- При успехе: клиент добавлен в очередь; при освобождении места получит push.
- При ошибке: состояние очереди не изменилось.

## UC-6: Оценка инструктора

**Актёры:** Клиент
**Предусловия:** У клиента есть бронь в статусе `attended` (факт посещения подтверждён бэкендом).
**Триггер:** Прошло ≥ 1 часа с окончания тренировки (FR-57).

**Бизнес-решения:**
- Только для посещённых (BR-11, FR-57). Для статусов `cancelled_late`, `no_show`, `cancelled_by_client`, `cancelled_by_gym` запрос не приходит.
- Только звёзды (BR-11, FR-58). Текстовых комментариев нет.
- Приватно для владельца. Клиентам публичный рейтинг не показывается.

**Основной поток:**
1. Через 1 час после окончания тренировки сервер инициирует push-уведомление с просьбой оценить инструктора (FR-57).
2. Клиент открывает push → веб-приложение открывается на экране оценки конкретной тренировки.
3. Клиент ставит оценку от 1 до 5 звёзд.
4. Клиент нажимает «Отправить».
5. Сервер сохраняет оценку; она видна только владельцу в админке.
6. Веб-приложение показывает подтверждение «Спасибо за оценку».

**Альтернативные потоки:**
- **A1. Клиент отложил оценку.** Push можно проигнорировать; запрос повторится при следующем открытии веб-приложения (в разделе «Мои бронирования» рядом с посещённой тренировкой без оценки будет CTA «Оценить»).
- **A2. Изменение оценки.** Если клиент уже оценил — повторный запрос возвращает текущую оценку с возможностью изменить.

**Исключения / ошибки:**
- **E1. Бронь не в статусе `attended`.** Сервер вернул `not_attended` (403) → оценка не принимается, показывается сообщение.
- **E2. Оценка уже отправлена и окно закрыто.** Сервер вернул `rating_window_closed` → показывается «Окно оценки закрыто».
- **E3. Сетевой сбой.** Клиент повторяет запрос с тем же `Idempotency-Key` (NFR-27).

**Постусловия:**
- При успехе: оценка сохранена на сервере.
- При ошибке: оценка не сохранена.

## UC-7: Управление профилем и выход из аккаунта

**Актёры:** Клиент
**Предусловия:** Клиент авторизован.
**Триггер:** Клиент хочет изменить свои данные или выйти из аккаунта.

**Бизнес-решения:**
- Редактирование профиля (FR-47). Имя, телефон, возраст, день рождения.
- Выход из аккаунта (FR-49, NFR-18). Локальная очистка сессионных токенов (из HttpOnly cookies / localStorage).

**Основной поток (редактирование):**
1. Клиент открывает раздел «Профиль».
2. Видит текущие данные: имя, телефон (маскированный, NFR-20), возраст, день рождения, бейдж «Постоянный клиент» (если `is_permanent=true`, FR-59).
3. Клиент редактирует поля и нажимает «Сохранить».
4. Сервер обновляет профиль.

**Основной поток (выход из аккаунта):**
1. Клиент в разделе «Профиль» нажимает «Выйти» (FR-49).
2. Веб-приложение очищает сессионные токены (из HttpOnly cookies / localStorage) (NFR-18).
3. Клиент переводится на экран входа.
4. Профиль и данные сохраняются на сервере — при следующем входе по тому же номеру клиент увидит всю историю.

**Альтернативные потоки:**
- **A1. Смена номера телефона.** Клиент пишет владельцу в Telegram (FR-16a). Владелец вручную привязывает новый номер через существующую админку (вне скоупа клиентского веб-приложения).

**Исключения / ошибки:**
- **E1. Сетевой сбой при сохранении профиля.** Клиент повторяет запрос; при успехе — профиль обновляется.
- **E2. Сетевой сбой при выходе.** Локальная очистка токенов выполняется в браузере независимо от сети; серверная сессия (refresh-токен) остаётся активной до истечения TTL.

**Постусловия:**
- При редактировании: профиль обновлён.
- При выходе: локальные токены очищены, клиент на экране входа; профиль и история броней сохранены на сервере.

## UC-8: Получение push-уведомлений

**Актёры:** Клиент (пассивный), Система (активный инициатор)
**Предусловия:** Клиент авторизован; push-подписка зарегистрирована на бэкенде; клиент не отозвал разрешение на push в браузере.
**Триггер:** Наступило событие, требующее уведомления клиента.

**Бизнес-решения:**
- Канал — только браузерный push через Web Push API (BR-12, NFR-17). SMS/email не используются.
- Тайминг напоминаний (NFR-17). За 24 ч и за 1 ч до старта.
- Graceful degradation (NFR-28). При недоступности push-сервисов — допустимая деградация, бронирование работает.

**Сценарии уведомлений:**
- **S1. Напоминание о тренировке (BR-12).** За 24 ч и за 1 ч до старта → push с деталями брони.
- **S2. Отмена тренировки скалодромом (BR-14, FR-46).** Владелец отменил слот в админке → всем клиентам с активными бронями приходит push с причиной и кнопкой «Выбрать другое время» (deep-link в расписание с фильтром по формату). Бронь переходит в статус `cancelled_by_gym`. Повторная запись на этот слот запрещена.
- **S3. Освобождение места из Alert List (BR-6, FR-61).** Первый клиент в очереди получает push «Появилось место на тренировке N» с deep-link в карточку слота.
- **S4. Запрос оценки инструктора (BR-11, FR-57).** Через 1 час после окончания тренировки — push с просьбой оценить.

**Основной поток:**
1. Сервер инициирует push через Web Push API (VAPID-подпись, service worker).
2. Клиент получает push в браузере.
3. Клиент нажимает на push → веб-приложение открывается на соответствующем экране (бронь / расписание с фильтром / карточка слота / экран оценки).

**Альтернативные потоки:**
- **A1. Push запрещён на уровне браузера (FR-60).** Веб-приложение один раз показывает предупреждение «Вы не будете получать напоминания, включите уведомления в настройках браузера». Функционал не блокируется; клиент узнаёт о событиях при открытии веб-приложения (баннер в «Мои бронирования» для отмен скалодромом и Alert List).
- **A2. Push-сервисы недоступны (NFR-28).** Уведомления не доставляются — допустимая деградация. Бронирование и отмена работают. «Потерянные» уведомления не отправляются повторно (MVP-упрощение).
- **A3. Клиент не нажал на push.** Информация о событии доступна в «Мои бронирования» при следующем открытии веб-приложения.

**Исключения / ошибки:**
- **E1. Невалидная push-подписка.** Сервер получил ошибку доставки → удаляет подписку из профиля клиента; новые уведомления не отправляются до повторной регистрации.
- **E2. Deep-link не сработал.** Веб-приложение открывается на главном экране; клиент находит нужную бронь/слот вручную.

**Постусловия:** Клиент информирован о событии (или узнаёт о нём при следующем открытии веб-приложения, если push не доставлен).
